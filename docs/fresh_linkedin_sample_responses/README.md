## Endpoint mapping (8 of 8 confirmed)

| Slug | Endpoint | Purpose |
|------|----------|---------|
| `company_by_url` | `GET /get-company-by-linkedinurl?linkedin_url=<company URL>` | Company profile by LinkedIn company URL |
| `company_by_domain` | `GET /get-company-by-domain?domain=<domain>` | Company profile by web domain (fuzzy — may return a regional variant) |
| `profile_posts` | `GET /get-profile-posts?linkedin_url=<profile URL>` | Recent posts by a person |
| `company_posts` | `GET /get-company-posts?linkedin_url=<company URL>` | Recent posts by a company |
| `post_reactions` | `GET /get-post-reactions?urn=<bare activity id>` | Reactors on a post |
| `post_comments` | `GET /get-post-comments?urn=<bare activity id>` | Comments on a post |
| `post_details` | `GET /get-post-details?urn=<bare activity id>` | Single post detail by URN |
| `search_posts` | `POST /search-posts` (JSON body) | Search posts across LinkedIn with rich filters — keyword, date window, content type, author/mentioner URNs, author industry/keyword; paginated by `page` |

## URN format

- The post `urn` field in `profile_posts` / `company_posts` is a bare activity id (e.g. `7450415215956987904`).
- This is the value required by `/get-post-reactions`, `/get-post-comments`, and `/get-post-details`. Passing `urn:li:activity:...`, a full `post_url`, or a `linkedin_url` returns HTTP 400.

## Resolved

All 8 endpoints are now mapped. `/search-posts` required POST with a JSON body (not the GET probes tried earlier — that's why the initial sweep missed it).
